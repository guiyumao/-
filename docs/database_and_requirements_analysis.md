# 实验室设备与耗材管理系统需求分析文档

本文档用于说明系统的业务目标、角色协同、用例设计、样例来源和数据约束。文档中的样例均直接来源于后端初始化文件，不使用硬编码演示值。

## 一、系统概览

系统面向高校实验室场景，覆盖设备、耗材、危化品、库存、借还、维修、校准、提醒、报表和权限控制等能力。

技术栈：
- 后端：Spring Boot 3、Spring Security、JWT、MyBatis-Plus
- 前端：Vue 3、TypeScript、Element Plus、ECharts、Vite
- 数据库：MySQL 8
- 缓存：Redis 7

系统入口：

| 入口 | 地址 |
|------|------|
| 前端 | `http://127.0.0.1:5173` |
| 后端 API | `http://127.0.0.1:8080/api` |
| Swagger 文档 | `http://127.0.0.1:8080/swagger-ui.html` |

## 二、角色与协同

系统默认启用 4 类角色：系统管理员、实验室主任、教师、学生。

| 角色 | 角色编码 | 主要职责 |
|------|----------|----------|
| 系统管理员 | `sys_admin` | 管理用户、角色、菜单权限、系统配置和审计日志 |
| 实验室主任 | `lab_director` | 统筹本实验室设备、耗材、危化品、安全、审批、提醒处理和统计数据 |
| 教师 | `teacher` | 发起设备借用、耗材申领、危化品使用、故障上报等申请，并为学生申请提供来源说明 |
| 学生 | `student` | 在授权范围内提交借用和申领申请，使用过程接受教师或实验室主任监管 |

协同原则：
- 教师和学生负责发起申请，申请记录必须带有用途、来源说明和责任人信息。
- 实验室主任负责审批、核验、异常转办和提醒处理。
- 系统管理员负责全局权限和数据治理，不替代业务来源责任人。
- 任一业务记录都应能追溯到申请人、审批人、执行人、提醒接收人和来源。
- 借用、维修、校准、提醒之间必须联动，不能只保留单模块状态。

## 三、功能需求

| 模块 | 功能需求 |
|------|----------|
| 登录认证 | 用户名密码登录、JWT 鉴权、登录状态恢复 |
| 权限管理 | 角色、菜单、按钮权限控制 |
| 用户管理 | 用户新增、编辑、停用、角色分配、密码重置 |
| 实验室管理 | 实验室基础信息维护、状态管理、负责人绑定 |
| 设备管理 | 设备分类、设备台账、设备状态、校准周期维护 |
| 设备借用 | 借用登记、审批关联、归还登记、状态更新、逾期催还、归还异常转维修 |
| 设备维修 | 报修、维修状态更新、费用和结果登记、借用异常来源关联、维修后回传校准 |
| 设备校准 | 校准任务、证书编号、校准结果和有效期维护、到期提醒联动、借用可用性控制 |
| 耗材管理 | 耗材分类、耗材台账、安全库存和有效期维护 |
| 耗材入库 | 批次、数量、单价、供应商、生产日期、有效期、来源单位记录 |
| 耗材出库 | 库存批次选择、出库数量校验、金额计算、申请人/审批人/操作人链路记录 |
| 危化品管理 | CAS 编号、危险类别、MSDS 编号、储存位置维护 |
| 危化品使用 | 入库、领用、归还、废液处理、余量记录、双人复核和教师来源绑定 |
| 库存管理 | 统一库存、批次库存、安全库存预警、过期预警 |
| 通知提醒 | 校准到期、耗材过期、设备逾期催还通知、待办责任人提示、来源链路展示 |
| 报表统计 | 设备借用、维修、校准、耗材出库、危化品使用统计 |
| 审计日志 | 关键业务操作记录和追踪 |

### 来源字段要求

1. 申请类数据至少记录申请人、用途、业务日期、来源说明。
2. 审批类数据至少记录审批人、审批结果、审批时间或审批依据。
3. 执行类数据至少记录操作人、执行时间、执行结果。
4. 提醒类数据至少记录发送人、接收人、关联业务类型、关联业务主键。
5. 批次类数据至少记录来源单位、批号、数量、时间和去向。

## 四、用例说明

系统用例按照“申请人 -> 审批人 -> 执行人/核验人 -> 提醒接收人”的链路组织。所有样例都直接对应后端样本数据。

### 1 设备借用与归还

| 项目 | 内容 |
|------|------|
| 用例名称 | 设备借用与归还 |
| 参与者 | 教师、学生、实验室主任、维修人员 |
| 前置条件 | 1. 申请人已登录并具备借用权限；<br>2. 系统已维护实验室、设备分类和设备台账；<br>3. 设备状态为“可用”；<br>4. 申请记录填写用途、时间和来源说明。 |
| 后置条件 | 1. 生成包含申请人、审批人、设备和来源说明的借用记录；<br>2. 审批通过后设备状态更新为“借用中”；<br>3. 归还核验完成后设备恢复“可用”或转入维修；<br>4. 逾期时生成可追溯催还通知。 |
| 基本事件流 | 1. 教师或学生进入设备借用页面查看设备名称、状态、校准信息和实验室归属。<br>2. 申请人填写借用用途、借用时间、应还时间和来源说明。<br>3. 系统校验设备状态和时间合法性。<br>4. 实验室主任查看申请内容并完成审批。<br>5. 审批通过后系统写入借用记录并更新设备状态。<br>6. 借用期间若接近到期或已逾期，系统向申请人发送催还通知。<br>7. 归还时由实验室主任或管理员录入归还检查结果。<br>8. 若归还异常，则同步转入设备维修流程。 |
| 用例关系 | 包含“设备状态校验”“审批处理”“归还登记”；扩展“逾期提醒”“设备维修”“设备校准可用性控制”。 |

### 4 耗材出库

| 项目 | 内容 |
|------|------|
| 用例名称 | 耗材出库 |
| 参与者 | 教师、学生、实验室主任、耗材管理员 |
| 前置条件 | 1. 参与者已登录；<br>2. 系统已维护耗材台账和库存批次；<br>3. 出库申请人具备申领权限；<br>4. 库存批次可用且未超出安全库存。 |
| 后置条件 | 1. 出库记录保留申请人、审批人、操作人和批次信息；<br>2. 库存批次数量完成扣减；<br>3. 金额和去向记录可追溯；<br>4. 库存不足或临期时产生提醒。 |
| 基本事件流 | 1. 申请人进入耗材出库页面。<br>2. 选择耗材、批次并填写用途。<br>3. 系统校验库存数量和出库数量。<br>4. 审批人确认后系统生成出库记录。<br>5. 操作人完成发放，系统同步扣减库存。<br>6. 出库记录可直接回溯到教师或学生申请样例。 |
| 用例关系 | 包含“库存批次校验”“审批处理”；扩展“库存预警”“报表统计”。 |

### 5 危化品使用

| 项目 | 内容 |
|------|------|
| 用例名称 | 危化品使用 |
| 参与者 | 教师、学生、实验室主任、危化品管理员、复核人 |
| 前置条件 | 1. 申请人已登录；<br>2. 系统已维护危化品台账和库存批次；<br>3. 学生申请时需绑定指导教师；<br>4. 库存数量和安全状态满足领用条件。 |
| 后置条件 | 1. 领用、归还或废液处理记录保存成功；<br>2. 对应库存批次数量完成更新；<br>3. 申请人、审批人、操作人和复核人信息完整保留。 |
| 基本事件流 | 1. 申请人进入危化品使用页面。<br>2. 选择危化品、批次和业务动作类型。<br>3. 填写使用数量、用途、项目名称和备注。<br>4. 实验室主任审核库存、安全培训和用途。<br>5. 审批通过后由操作人和复核人完成登记。<br>6. 使用结束后登记剩余量、包装回收或废液处理。 |
| 用例关系 | 包含“库存批次校验”“审批处理”“双人复核”；扩展“废液处理”“安全审计”。 |

### 6 设备维修

| 项目 | 内容 |
|------|------|
| 用例名称 | 设备维修 |
| 参与者 | 教师、学生、实验室主任、维修人员 |
| 前置条件 | 1. 报修人已登录系统；<br>2. 系统已存在对应设备台账；<br>3. 设备未处于报废状态；<br>4. 报修记录说明故障来源。 |
| 后置条件 | 1. 系统生成包含报修人、处理人和来源说明的维修记录；<br>2. 设备状态更新为“维修中”或“无法修复”；<br>3. 维修结果回传到设备台账，并可继续转入校准；<br>4. 与借用相关的维修能反向追踪到原借用链路。 |
| 基本事件流 | 1. 报修人进入设备维修页面，选择故障设备。<br>2. 填写故障描述、报修时间和来源说明。<br>3. 实验室主任确认报修并分派维修人员。<br>4. 维修人员录入过程、费用、结果和结束时间。<br>5. 若该报修来自借用归还异常，系统保留与借用记录的关联。<br>6. 维修完成后可继续进入校准流程。 |
| 用例关系 | 包含“处理人分派”“维修结果登记”；扩展“设备借用异常转维修”“设备校准”“系统日志”。 |

### 7 设备校准

| 项目 | 内容 |
|------|------|
| 用例名称 | 设备校准 |
| 参与者 | 实验室主任、校准人员 |
| 前置条件 | 1. 参与者已登录系统；<br>2. 系统已存在设备台账和校准周期配置；<br>3. 校准任务来源于年度计划、维修复检或提醒到期；<br>4. 校准记录填写证书编号、日期和来源说明。 |
| 后置条件 | 1. 设备校准记录保存成功；<br>2. 设备状态和下次校准日期更新；<br>3. 校准结果影响设备借用可用性；<br>4. 到期前再次进入提醒链路。 |
| 基本事件流 | 1. 实验室主任进入设备校准页面。<br>2. 填写证书编号、校准日期、有效期和来源说明。<br>3. 系统校验证书编号唯一，且有效期不得早于校准日期。<br>4. 保存后设备状态更新为“校准中”。<br>5. 校准确认人录入校准结果。<br>6. 若结果合格，系统更新下次校准日期并恢复设备可用性。<br>7. 若结果不合格，系统限制借用并提示继续处理。<br>8. 到期窗口内的设备自动进入提醒中心。 |
| 用例关系 | 包含“证书编号校验”“校准结果登记”；扩展“到期提醒”“设备维修”“设备借用可用性控制”。 |

### 8 删除类用例

| 项目 | 内容 |
|------|------|
| 用例名称 | 业务记录删除与追溯 |
| 参与者 | 具备编辑权限的教师、实验室主任、管理员 |
| 适用对象 | `equipment-borrows`、`consumable-outbounds`、`hazardous-usages`、`equipment-calibrations`、`equipment-repairs` |
| 前置条件 | 1. 用户已登录且拥有编辑权限；<br>2. 被删除对象属于业务记录而非基础台账；<br>3. 当前记录未被审计或归档规则锁定。 |
| 后置条件 | 1. 系统删除对应业务记录；<br>2. 列表与统计结果同步刷新；<br>3. 操作结果进入审计链路；<br>4. 关联提醒与报表保留历史痕迹。 |
| 基本事件流 | 1. 用户在记录卡片或列表中点击“删除”。<br>2. 系统弹出二次确认，提示删除影响范围。<br>3. 用户确认后，系统调用后端删除接口。<br>4. 删除完成后页面自动刷新。<br>5. 若记录已关联提醒或统计，系统保留历史追溯信息，不删除来源样本。 |
| 删除接口 | `DELETE /api/equipment-borrows/{id}`、`DELETE /api/consumable-outbounds/{id}`、`DELETE /api/hazardous-usages/{id}`、`DELETE /api/equipment-calibrations/{id}`、`DELETE /api/equipment-repairs/{id}` |
| 用例关系 | 删除只作用于业务流水，不影响设备、耗材、危化品等基础台账。 |

### 9 到期提醒

| 项目 | 内容 |
|------|------|
| 用例名称 | 到期提醒 |
| 参与者 | 系统管理员、实验室主任、教师 |
| 前置条件 | 1. 系统已配置提醒规则；<br>2. 存在待提醒的校准、库存或借用记录；<br>3. 用户已登录并具备查看提醒权限。 |
| 后置条件 | 1. 页面展示当前用户可见的提醒与通知数据；<br>2. 每条提醒都能看到责任角色、来源类型和关联业务；<br>3. 即将到期的校准和耗材批次形成待办；<br>4. 借用逾期提醒可追踪发送人、接收人和业务主键。 |
| 基本事件流 | 1. 用户进入提醒中心。<br>2. 系统加载“我的通知”“校准协同待办”“耗材批次待办”等数据。<br>3. 系统筛选未来阈值天数内即将到期的设备校准记录。<br>4. 系统筛选未来阈值天数内即将过期的耗材库存批次。<br>5. 若借用业务中存在逾期记录，实验室主任可发送催还通知。<br>6. 用户点击刷新后，系统重新拉取待办和通知数据。 |
| 用例关系 | 包含“用户登录”“提醒规则计算”“通知列表查询”；扩展“设备借用逾期催还”“设备校准”“耗材库存预警”“设备维修后校准跟进”。 |

## 六、数据约束

- 申请类数据至少记录申请人、用途、业务日期和来源说明。
- 审批类数据至少记录审批人、审批结果和审批时间或依据。
- 执行类数据至少记录操作人、执行时间和执行结果。
- 提醒类数据至少记录发送人、接收人、关联业务类型和主键。
- 批次类数据至少记录来源单位、批号、数量、时间和去向。

## 七、已实现接口

业务删除接口已经在前后端接通：
- `equipment-borrows` 删除
- `consumable-outbounds` 删除
- `hazardous-usages` 删除
- `equipment-calibrations` 删除
- `equipment-repairs` 删除

这些删除操作只面向业务流水，不会删除设备、耗材、危化品等基础台账。
